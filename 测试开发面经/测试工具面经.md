### 在 Shell 中查找名为 `A` 且进程号为 `111` 的进程所在目录的示例命令：
```ps -ef | grep "A" | grep "111" | awk '{print $7}'```
- `ps -ef` ：显示系统中所有进程的详细信息。
- `grep "A"` ：筛选出包含字符串 `A` 的进程行。
- `grep "111"` ：在上述结果中进一步筛选出进程号为 `111` 的行。
- `awk '{print $7}'` ：提取出第七个字段，通常是进程的工作目录。
### 3、git命令
**1. `git init`**：  
初始化一个新的 Git 仓库。
**2. `git clone <repository>`**：  
从远程仓库克隆到本地。
**3. `git add <file>` 或 `git add.`**：  
将文件添加到暂存区。
**4. `git commit -m "message"`**：  
提交暂存区的更改，并添加描述信息。
**5. `git status`**：  
查看工作区和暂存区的状态。
**6. `git branch`**：  
查看本地分支。
**7. `git checkout <branch>`**：  
切换分支。
**8. `git merge <branch>`**：  
将指定分支合并到当前分支。
**9. `git pull`**：  
从远程仓库拉取最新的更改并合并到本地。
**10. `git push`**：  
将本地的提交推送到远程仓库。
**11. `git log`**：  
查看提交历史。
**12. `git diff`**：  
查看工作区和暂存区的差异，或者不同提交之间的差异。

### 4、在本地分支做了修改，怎么提交到远程分支上，并且不会引起冲突
1. 首先，确保您已经将修改的文件添加到暂存区：
```
   git add <your_modified_files>
```
2. 提交您的更改，并提供有意义的提交消息：
```
   git commit -m "Your descriptive commit message"
```
3. 在推送更改之前，先拉取远程分支的最新代码，以确保您的本地分支与远程分支是同步的，并解决可能出现的冲突
```
   git pull origin <remote_branch_name>
```
如果有冲突，您需要手动解决冲突，然后再次添加和提交解决后的更改。
4. 推送您的本地分支到远程分支：
```
   git push origin <your_local_branch_name>
```

### 5、docker命令
**1. `docker version`**：  
查看 Docker 版本信息。
**2. `docker info`**：  
显示 Docker 系统的各种信息，如容器和镜像数量、存储驱动等。
**3. `docker search <image_name>`**：  
在 Docker Hub 上搜索指定名称的镜像。
**4. `docker pull <image_name>`**：  
从 Docker Hub 拉取指定的镜像。
**5. `docker images`**：  
列出本地已有的镜像。
**6. `docker run <image_name>`**：  
基于指定镜像创建并启动一个新容器。
- 例如：`docker run -d -p 8080:80 nginx` 以守护式（后台）运行 Nginx 容器，并将主机的 8080 端口映射到容器的 80 端口。
**7. `docker ps`**：  
列出正在运行的容器。
**8. `docker ps -a`**：  
列出所有容器，包括已停止的。
**9. `docker stop <container_id>`**：  
停止指定的容器。
**10. `docker start <container_id>`**：  
启动已停止的容器。
**11. `docker restart <container_id>`**：  
重新启动指定的容器。
**12. `docker rm <container_id>`**：  
删除指定的容器。
**13. `docker rmi <image_id>`**：  
删除指定的镜像。
**14. `docker exec -it <container_id> <command>`**：  
在运行中的容器内执行指定命令。
- 例如：`docker exec -it my_container bash` 进入名为 `my_container` 的容器并打开 Bash 终端。
**15. `docker logs <container_id>`**：  
查看指定容器的日志输出。
**16. `docker build -t <image_name> <directory>`**：  
使用指定目录中的 Dockerfile 构建一个新的镜像，并指定镜像名称。


### 6、用K8S做什么
我们的后端服务是微服务架构，各服务组件是以容器的形式部署到K8S集群上，K8S帮助进行资源管理和分配、负载均衡，测试在工作中接口有问题，会去看一下POD的运行情况，或去重启下POD实例。还会在K8S中调用接口、定时任务调用和debug，debug：当应用出现异常时，首先通过kubectl logs  查看相关 Pod 的日志，确定是否有错误信息。如果需要进一步排查，使用 kubectl exec 进入 Pod 内部执行相关命令查看环境和配置情况。


### 7、怎么去查看接口日志
我们的服务产生日志后，会通过filebeat监控日志变化并发送数据，数据进入kafka缓冲区，logstach会过滤和格式化数据，给日志打一些标签，日志最终进入elasticsearch数据库，kibana是一个可视化界面，可以查询日志。

### 1、如何设计测试策略
理解系统：了解系统的功能、架构、技术细节和用户需求，全面把握的系统的复杂度和测试需求
测试分类：根据系统的功能和复杂性，对测试分类（可以利用测试四象限、金字塔分层)，每种测试关注的测试目标和风险不一致。如：单元测试、功能测试、接口测试、探索性测试、UAT回归测试、性能测试
确定测试重点：针对每种分类，确定关键的测试重点。比如功能测试中重点是核心业务场景，回归测试是流程的完整可用
制定测试策略：根绝测试分类和测试重点，制定测试的范围、方法、测试技术、测试工具。
规划测试活动：将测试策略转化为实际的测试活动和计划。确定测试策略的时间安排、资源分配、测试用例设计和执行的细节
执行和监督测试：执行测试活动并监控测试流程和结果
完善测试策略: 根据反馈，完善测试策略。

### 1. How to Design a Testing Strategy
Understand the System: Comprehend the functions, architecture, technical details and user requirements of the system to fully grasp the complexity and testing requirements of the system.  
Testing Classification: Based on the functions and complexity of the system, classify the tests (the test quadrants and pyramid layering can be utilized). Each type of test has different testing goals and risks. Such as: unit testing, functional testing, interface testing, exploratory testing, UAT regression testing, performance testing.  
Determine Testing Focus: For each classification, determine the key testing focuses. For example, in functional testing, the focus is on core business scenarios, and in regression testing, it is the complete usability of the process.  
Formulate the Testing Strategy: Based on the testing classification and testing focuses, formulate the testing scope, methods, testing techniques and testing tools.  
Plan the Testing Activities: Transform the testing strategy into actual testing activities and plans. Determine the time arrangement, resource allocation, details of test case design and execution of the testing strategy.  
Execute and Monitor the Tests: Execute the testing activities and monitor the testing process and results.  
Improve the Testing Strategy: Based on the feedback, improve the testing strategy.

### 6、单点登录
1. **基于令牌（Token）的方式**：
    - 用户在认证服务器进行登录，认证成功后，服务器生成一个加密的令牌（Token）并返回给用户。
    - 用户带着令牌访问其他应用系统，应用系统通过与认证服务器通信验证令牌的有效性。
2. **基于 Cookie 的方式**：
    - 用户在认证服务器登录成功后，认证服务器设置一个特定的 Cookie，该 Cookie 包含登录凭证信息。
    - 当用户访问其他应用系统时，应用系统检查该 Cookie，如果存在且有效，则认为用户已登录。
### 7、在项目中参与哪些质量保障(quality assurance)工作
回答这个问题，首先需要我来说下，对于质量保障工作的理解，软件的高质量不是测试出来的，而是开发和设计出来的，所以这不仅要求我们对交付的(delivered)软件进行测试，对各个阶段产物的评审、QA对流程的监控，对测试计划和策略的合理配置(reasonable configuration)。所以接下来我说的质量保证工作，主要针对各级别测试活动（包括自动化测试）之外的工作进行展开。
1、设计测试策略和测试计划，确定测试的范围、资源和进度(progress)。
2、需求评审，从测试角度发现需求遗漏
3、风险评估(conduct risk assessment)，识别风险(identity risk)，制定解决方案，协调资源（coordinate resource)（识别到卡片依赖度高identify the high dependence on card，前期开发进度滞后lag，反馈团队，调整卡片开发优先级adjust the development priority，调整后期测试计划）
4、质量跟踪和度量，缺陷defects分析，提供质量报告，及时与团队同步synchronize with the team
5、优化流程，输出开解卡规范文档output specification documents of kick in a kick off card activity、整理常见web端交互场景 sort out common web interaction scenarios等
### 9、web端测试、APP端测试和接口测试的区别
web端主要放在浏览器，会考虑主流浏览器edge、chrome、Safari等和无界面浏览器，也会看分辨率
APP端浏览器会涉及到下载、卸载功能，应用会有移动端和平板端，考虑Android、iPhone、鸿蒙，也会考虑不同分辨率的设备，下载方式也有区别，可能是在应用市场上、官网下载等
接口主要测试接口的连通性，会去测试参数、header、data的组合场景

### 2、如何解决接口数据依赖问题，请举例说明？
数据依赖有这几种解决方式：  
第一种，global全部变量的方式解决，将你需要用到的依赖数据提取出来，升他为全局变量。比如说在APP接口测试里面常用到token做鉴权效验，用一个全局变量来处理依赖的数据，比如登录后返回token,其它接口都需要这个token,那就用全局变量来传token参数。全局变量有其局限性，不能夸.py文件调用。

具体参考我这篇博客：[https://www.cnblogs.com/xiamaojjie/p/12031265.html](https://www.cnblogs.com/xiamaojjie/p/12031265.html)

第二种，放setUp里面，将接口有依赖的数据提取出来，放到setUp里面，每次运行用例前，都会首先自动运行setUp函数。

具体参考我这篇博客：[https://www.cnblogs.com/xiamaojjie/p/12034422.html](https://www.cnblogs.com/xiamaojjie/p/12034422.html)

第三种，反射实现。首先定义一个类，比如说获取cookie，定义一个GetCookie，里面的类属性给他一个初始值，比如说cookie=None。然后用python里面的setattr函数给刚才定义cookie值为空的属性赋值。最后再用getattr函数取出里面的cookie。个人感觉这种方式实现比较好，也不用担心cookie有效期的问题。
另一个例子：当测试需要根据不同的环境（如开发、测试、生产）加载不同的配置数据时，可以使用反射来读取和处理这些动态配置。

具体参考我这篇博客：[https://www.cnblogs.com/xiamaojjie/p/12032199.html](https://www.cnblogs.com/xiamaojjie/p/12032199.html)

### 4、如何实现用例可配置（可配置指的是实现运行全部用例和只运行部分用例）？
假如每一个测试用例都有一个case_id，可以创建以.config文件结尾的配置文件，里面指定mode和部分运行的case_id, 写一个文件来读取（配置文件要和读取配置文件的方法放在一起，否则会报文件找不到的错误），判断mode如果等于all，就执行全部用例。如果mode等于列表，就判断传入列表是否在case_id中，最后在执行要执行的用例。

### 5、在接口自动化中有遇到过什么样的问题？

接口之间存在复杂的数据依赖关系，处理不当会导致测试结果不准确。例如，一个接口的输出是另一个接口的输入，如果前一个接口的结果异常，可能影响后续接口的测试。

### 7、requests中，传递参数params，data，json的区别是？
在 `requests` 库中，`params`、`data` 和 `json` 这三个参数用于向服务器传递不同类型的数据
**`params`**：
- 通常用于传递查询参数，添加到 URL 的问号后面。
- 例如，如果要发送 `page=1` 和 `limit=10` 这样的查询参数
```python
params = {'page': 1, 'limit': 10} 
response = requests.get('http://example.com/api', params=params)
```
**`data`**：
- 用于发送表单数据，-数据会被编码为 `application/x-www-form-urlencoded` 格式。
```python
data = {'username': 'user1', 'password': 'pass1'}
response = requests.post('http://example.com/login', data=data)
```
**`json`**：
- 用于发送 JSON 格式的数据，`requests` 库会自动将 Python 对象编码为 JSON 格式，并设置正确的 `Content-Type` 为 `application/json`，当数据具有更结构化和层次化的特点，适合用 JSON 来表示时。
```python
json_data = {
    "name": "John Doe",
    "age": 25,
    "address": {
        "street": "123 Main St",
        "city": "Anytown",
        "state": "CA"
    }
}
response = requests.post('http://example.com/create_user', json=json_data)
```
具体使用哪种方式取决于服务器的接口要求和您要发送的数据类型。
### 8、项目中有没有加入日志信息，怎么实现的？
给用例添加日志信息，用logging这个Python自带库，用例通过时不会显示捕获日志，用例失败时才会显示，根绝info,debug,warning,error,critical针对性的写了一个日志类，设置好日志输出格式，将日志指定保存到.txt结尾的文本文件里面，像一些常见的打印信息，就用info收集，在可能会出现错误的try-except结构中就用error收集错误信息，方便定位问题。
日志级别：NOTSET<DEBUG < INFO < WARNING < ERROR < CRITICAL
```python
import logging

def test_demo():
	logging.info("this is warning log ...")
    logging.warning("this is warning log ...")
    logging.error("this is error log ...")
    logging.critical("this is critical log ...")
    assert 1==2
```

### 9、有没有做过接口加密之类的接口？
比如用户登录认证接口使用了加密算法，在设计自动化时，如果要验证加密接口响应值正确性的话，就必须使用正确的解密方式先对其实现解密，再完成验证。
**场景**：QA不了解对应的加密算法，需要研发提供加解密对应的 lib 包，完成数据解密。
 **设计过程**：1. 调用python自带的base64，直接对返回的响应做解密，获取到正常的数据再进行断言
```python
import requests
import base64
#加密
secret_msg=base64.b64encode("用户身份证",encode('utf-8'))
def test_end():
	url="https://httpbin.ceshiren.com/post"
	data={"msg":secret_msg}
	#发送请求接口
	res=requests.post(url,data=data)
	#获取加密的响应数据
	msg=res.json()["form"]["msg"]
	#对获取的加密数据进行解密
	encoded_str=base64.b64decode(msg).decode('utf-8')
	assert encoded_str=="山东德州"
 
```

### 11：接口测试发现哪些bug？
通过mock 边缘的/业务难以生产的请求体，比如接口的权限验证，用一个已失效的token和一个被拒绝权限的token发出请求，返回body没有差别，前端显示界面一样。
实际上，在设计中前端应该展示“NO AUTHORITY”和“BE REJECTED"，需要在接口中返回message做区别处理。

### 14、token、session、cookie的区别
Token 用于身份验证，包含丰富的用户信息，并进行用加密和签名保证安全性，不需要服务器在内存存储
例子：用户生成成功后，服务器会生成一个token返回给客户端，客户端在后续请求时携带该token以证明身份

Session 是在服务器端保存的用户会话信息，有唯一标识，依赖于 Cookie 或 URL 重写来传递会话标识。
- 例子： 用户登录后，服务器创建一个 Session，并将会话标识通过 Cookie 发送给客户端。客户端后续请求时，服务器通过 Cookie 中的会话标识找到对应的会话信息。

Cookie 是存储在客户端（通常是浏览器）的一小段数据，客户端在后续请求时自动携带，有大小限制和域名限制， 可以设置过期时间。
- 例子：服务器设置一个记住用户名的 Cookie，下次用户访问时，浏览器自动发送该 Cookie，服务器可以获取到用户名信息。 


### 16、讲一下接口测试怎么做
大体流程：接口需求分析、接口测试工具选择、接口用例设计与执行、接口测试回归、接口测试自动化持续集成，接口测试流程大体分为：
一、分析接口需求，请Dev提供接口说明文档
二、从接口说明文档整理出接口测试用例，包括详细的入参（正常情况，异常情况包括输入参数个数，类型，可选/必选，考虑参数有互斥或关联）和出参数据（符合接口文档需求）以及明确的格式和检查点
三、与开发评审接口测试用例
四、准备入参、出参测试数据，整理成json/xml/excel等格式
五、结合接口测试用例文档和Excel格式的数据文档，编写接口自动化测试的业务逻辑代码；
六、开始执行接口自动化测试用例；
七、如有bug，提bug，开发修复，回归bug，跟踪状态
八、完成后进行自动化持续集成；
